用 Claude Code 搭建小型 Coding Agent：一次产品思维驱动的工程验证

查看源码 实验一session 实验二session

事情背景

我在研究 Claude Code、Codex CLI、OpenClaw 等产品的 runtime 机制后，形成了几个核心判断——上下文治理的本质是注意力管理而非省 token，工具的核心价值是环境过滤而非功能调用，模型能力和 harness 复杂度之间存在分层关系。但这些判断停留在"读完觉得有道理"的阶段，没有实验数据支撑。

遇到了什么障碍

要验证这些判断，需要搭一个 agent 原型、实现工具系统、设计对照实验、批量跑测试、收集 trace 并做跨模型对比。这整条工程闭环以前对我来说很难做到——从"有一个想法"到"拿到可分析的实验数据"中间隔着太多工程环节。纯靠在 LLM 对话里完成也不现实：项目涉及多个文件之间的调用依赖，上下文一长，文件间的接口细节就开始对不上；出了问题也很难定位，因为 LLM 没有办法自己跑测试、看报错、再回去改。

Claude Code 解决了这个问题。我只需要想清楚实验该怎么设计、变量怎么控制，和它对几轮确认思路，剩下的工程实现和测试执行都交给它在项目目录里直接完成。

第一轮：验证 harness 设计的核心判断

我搭了一个 Claude Code 风格的最小 agent 原型（tool loop + 上下文管理 + 6 个工具），在 workspace 里埋了一个跨文件 bug，然后做了两组 A/B 实验。

实验一：上下文压缩 ON vs OFF

结论清晰——OFF 组 3 次全部失败，agent 陷入死循环，20 步里 18 步在重复相同的失败操作；ON 组 3 次中 2 次成功，路径干净。messages 堆积导致有效信息密度下降，模型注意力被淹没，丧失了"退一步换策略"的能力。核心判断得到验证：上下文治理是注意力管理。

OFF-3 死循环

实验二：搜索过滤 filtered vs unfiltered

跨两个模型（moonshot-v1-8k / kimi-k2.5）做测试。弱模型两组都卡在 write_patch 的基础能力上，搜索过滤的差异没机会体现；强模型 filtered 组 3/3 成功，unfiltered 组 2/3 成功，唯一一次失败是信息过载引发的幻觉——read_file 失败后涌入 38,536 chars 的异常数据，模型随即声称已修复 bug 并给出详细"修复方案"，但实际上没有执行任何 write_patch。

结论：模型能力和 harness 复杂度是分层关系——弱模型的瓶颈在基础工具使用，强模型才到信息过滤这一层。不受控的信息涌入增加了幻觉风险，但不是必然导致失败。

但也正是这里出现了一个新的卡点：kimi-k2.5 太强，当前 bug 难度不够，无法区分模型能力的上限。

第二轮：升级为多模型 Benchmark

实验升级

针对第一轮的卡点，我做了三件事：将 bug 从简单的类型不匹配升级为跨模块共享状态污染（表面 traceback 指向 auth.py，但 auth.py 逻辑完全正确，根因在 routes.py 篡改了全局状态后未恢复）；加入完整的 trace 记录系统；将测试模型扩展到五个（Claude Opus 4.6、DS-V3.2 Chat、DS-V3.2 Reason、GLM5-Turbo、GPT 5.4），每个跑 5 次取分布。其中 DS-V3.2 的 Chat 和 Reason 两个版本共享底座模型，专门用于验证"推理更强是否意味着 agent 更强"。

核心发现与失败模式

Agent 能力 ≠ 推理能力。 这是最重要的结论。DS-V3.2 Reason 诊断正确率 100% 但完成率仅 20%；GPT 5.4 诊断正确率 80% 但完成率 0%。推理能力与任务完成率之间是非线性关系。Agent 能力至少由三个独立维度构成：跨文件重建因果链的推理能力、把方案可靠序列化为工具调用的执行能力、无需用户确认时持续推进的行为范式。三项缺一不可，且未表现出稳定正相关。

同底座模型，Chat 显著优于 Reason。 DS-V3.2 Chat 80% vs Reason 20%，差距 4 倍。Reason 更长的推理链不仅没带来更高完成率，反而导致格式冲突和执行幻觉。

现在还卡在哪

Chat vs Reason 的归因模糊

当前最核心的未解决问题是 Chat vs Reason 的归因模糊。

DS-V3.2 Reason 完成率只有 20%，表面上看是执行幻觉——模型推理到了正确答案却没有正确调用工具。但我的 system prompt 要求输出必须以 TOOL: 或 FINAL: 开头，这与 Reason 模型先输出内部推理链再输出行动的原生格式直接冲突。格式遵循本身确实是模型能力的一部分，但问题是：如果换一种对 Reason 更友好的格式约束（比如允许先输出 thinking 再输出 action），完成率会不会大幅提升？

如果会，那 20% 的低完成率有很大一部分是 harness 设计导致的，不是模型本身的 agent 能力不足。如果换了格式完成率依然很低，才能下"Reason 不适合 agent 场景"的结论。目前的实验数据无法区分这两种情况。

产品设计权衡

这个归因问题背后指向一个更深层的产品设计权衡：agent 的 runtime 应该适配模型的原生行为模式，还是要求模型适配统一的格式规范？前者意味着要为不同类型的模型设计不同的 parser，工程复杂度上升；后者意味着可能系统性地低估某类模型的真实能力。对于要支持多模型接入的 agent 产品来说，这不是一个可以回避的问题。

要解决这个卡点，需要设计一组控制了格式变量的对照实验：在对 Reason 模型更友好的格式约束下重新跑一轮 benchmark，与当前结果做对比。但"更友好"的格式约束本身怎么定义、如何保证与 Chat 版本的对比仍然公平，这是我还在思考的问题。
